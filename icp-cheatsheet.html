Canisters
Canisters — ICP's smart contract execution units
A canister is a WebAssembly module deployed to the Internet Computer. It bundles code + state together — unlike stateless lambdas, canisters remember their data between calls. Each canister runs on a subnet of nodes that reach consensus before committing state changes. Canisters communicate via messages, not shared memory. The ic_cdk crate provides macros and APIs to write Rust canisters. Every public method is tagged with a macro that determines its message type.
WebAssembly module persistent state consensus-based

⚠ Key gotchas
  • ▸Queries don't commit writes: any mutation inside a #[query] is silently discarded. If you need to write, use #[update].
  • ▸Trapping rolls back state: if your update call panics or calls ic_cdk::trap(), all state changes in that call are reverted — like a database transaction rollback.
  • ▸No true parallelism: a canister processes one message at a time. Between await points in async calls, other messages can be interleaved — be careful with state.
  • ▸Heartbeat costs cycles: #[heartbeat] runs every consensus round and drains your cycle balance. Keep it fast or use timers instead.
  • ▸Composite queries: since dfx 0.15, queries can call other queries with #[ic_cdk::query(composite = true)] — but they still can't call update methods.
Memory Model — heap vs stable memory
Canisters have two distinct memory spaces. Heap memory is the Wasm linear memory — fast, ergonomic Rust data, but wiped on upgrade. Stable memory persists across upgrades and is up to 500 GiB. The critical decision for any canister: which data lives where? The wrong choice means losing user data on every deploy.
two memory spaces heap = wiped on upgrade stable = persists forever
Critical: when you upgrade a canister, heap memory is completely wiped. Any data stored only in heap (regular Rust variables, thread_local state) is LOST unless you explicitly serialize it to stable memory in #[pre_upgrade] and restore it in #[post_upgrade].

⚠ Common mistakes
  • ▸Forgetting pre_upgrade: no hook + heap-only data = every dfx deploy wipes your data. Always consider upgrade safety from day 1.
  • ▸OOM during serialization: serializing heap to stable requires both to fit simultaneously. With large datasets, Strategy 2 is mandatory.
  • ▸Raw stable memory API: ic_cdk::api::stable::stable_read() and stable_write() exist but are low-level. Use ic-stable-structures instead.
thread_local! & RefCell — the canonical canister state pattern
Canisters compile to WebAssembly, which has no threads — but Rust's safety rules still apply. Global mutable state normally requires unsafe or Mutex. In canisters, the solution is thread_local! combined with RefCell. This works because: (1) Wasm is single-threaded so no real thread contention exists, and (2) thread_local! satisfies Rust's borrowing rules for statics without needing Sync. This is the canonical pattern for all mutable canister state.
canonical state pattern single-threaded Wasm no Mutex needed
Why thread_local! is needed
// WRONG — won't compile: RefCell is not Sync
static STATE: RefCell<u64> = RefCell::new(0); // error!

// WRONG — works but unsafe
static mut COUNT: u64 = 0; // never do this

// CORRECT — thread_local! wraps in LocalKey, no Sync required
thread_local! {
    static STATE: RefCell<u64> = RefCell::new(0);
}
// In Wasm there is only one "thread" — this is effectively a safe global
The with() / borrow() / borrow_mut() pattern
thread_local! {
    static USERS: RefCell<HashMap<u64, String>> = RefCell::new(HashMap::new());
    static COUNTER: RefCell<u64> = RefCell::new(0);
}

#[ic_cdk::query]
fn get_user(id: u64) -> Option<String> {
    USERS.with(|u| u.borrow().get(&id).cloned())
    // .with() — closes borrow at end of closure
}

#[ic_cdk::update]
fn add_user(id: u64, name: String) {
    USERS.with(|u| { u.borrow_mut().insert(id, name); });
    COUNTER.with(|c| *c.borrow_mut() += 1);
}
Grouping state + helper functions
#[derive(Default)]
struct State {
    users:  HashMap<u64, User>,
    admins: HashSet<Principal>,
    config: Config,
}

thread_local! {
    static STATE: RefCell<State> = RefCell::new(State::default());
}

// Helper functions — avoids repeating STATE.with(...)
fn read_state<R>(f: impl FnOnce(&State) -> R) -> R {
    STATE.with(|s| f(&s.borrow()))
}
fn mutate_state<R>(f: impl FnOnce(&mut State) -> R) -> R {
    STATE.with(|s| f(&mut s.borrow_mut()))
}

#[ic_cdk::query]
fn is_admin(p: Principal) -> bool {
    read_state(|s| s.admins.contains(&p))
}
⚠ Borrow panic traps
  • ▸Double borrow panics: calling borrow_mut() while borrow() is still active panics at runtime. Always close borrows before opening new ones.
  • ▸Borrow across await — the #1 async bug: holding a Ref/RefMut across an .await point causes a runtime panic. Extract the value first, drop the borrow, then await.
  • ▸Safe async pattern: let val = STATE.with(|s| s.borrow().field.clone()); /* borrow dropped */ some_future.await; mutate_state(|s| s.other = val);
  • ▸Cell<T> for Copy types: if state is a simple Copy type (u64, bool), use Cell<T> instead of RefCell<T> — no runtime borrow check overhead.
Stable Memory — ic-stable-structures
The ic-stable-structures crate provides B-tree maps, vectors, and cells that live directly in stable memory. No serialization on upgrade — data is always there. The MemoryManager virtualizes stable memory into named regions so multiple data structures coexist. Each structure gets a unique MemoryId. This is the modern, recommended approach for any production canister.
ic-stable-structures no upgrade serialization up to 500 GiB

MemoryManager initialisation
use ic_stable_structures::{
    memory_manager::{MemoryId, MemoryManager, VirtualMemory},
    DefaultMemoryImpl, StableBTreeMap, StableCell, StableVec,
};
use std::cell::RefCell;

type Memory = VirtualMemory<DefaultMemoryImpl>;

// Each structure gets a unique MemoryId (0–254)
const USERS_MEM:  MemoryId = MemoryId::new(0);
const CONFIG_MEM: MemoryId = MemoryId::new(1);
const LOGS_MEM:   MemoryId = MemoryId::new(2);

thread_local! {
    // Always initialize MemoryManager first
    static MM: RefCell<MemoryManager<DefaultMemoryImpl>> =
        RefCell::new(MemoryManager::init(DefaultMemoryImpl::default()));

    // Use ::init() NOT ::new() — init() opens existing stable data on upgrade
    static USERS: RefCell<StableBTreeMap<u64, UserData, Memory>> =
        RefCell::new(StableBTreeMap::init(
            MM.with(|m| m.borrow().get(USERS_MEM))
        ));

    static CONFIG: RefCell<StableCell<Config, Memory>> =
        RefCell::new(StableCell::init(
            MM.with(|m| m.borrow().get(CONFIG_MEM)),
            Config::default(),
        ).unwrap());
}

use std::borrow::Cow;
use candid::{CandidType, Decode, Encode};

#[derive(CandidType, serde::Deserialize, Clone)]
struct UserData {
    name:       String,
    balance:    u64,

impl Storable for UserData {
    fn to_bytes(&self) -> Cow<[u8]> {
        Cow::Owned(Encode!(self).unwrap())   // Candid encoding
    }
    fn from_bytes(bytes: Cow<[u8]>) -> Self {
        Decode!(bytes.as_ref(), UserData).unwrap()
    }
    // Unbounded = variable size (strings, nested types)
    // Bounded { max_size: N, is_fixed_size } = for u64, [u8;32] etc.
    const BOUND: Bound = Bound::Unbounded;
}

⚠ Stable structures gotchas
  • ▸Never change MemoryIds: once a MemoryId is used and data is written, it's permanent. Changing it corrupts data — the memory manager won't find it.
  • ▸init() vs new(): always use ::init() in thread_local!. ::new() wipes existing data on every canister restart.
  • ▸StableBTreeMap returns owned values: unlike HashMap, .get() returns a cloned T, not a reference — because the data lives in stable memory, not Rust's heap.
  • ▸Use Bound::Bounded for fixed-size types: u64, Principal, [u8;32] — mark them as Bounded for BTree node optimization.
Upgrades — surviving code deployments
Every dfx deploy on an existing canister performs an upgrade. The Wasm module is replaced, stable memory survives, heap memory does not. Sequence: #[pre_upgrade] runs → Wasm replaced → #[post_upgrade] runs. With ic-stable-structures, no serialize/deserialize hooks are needed for data persistence.
pre/post upgrade hooks stable memory survives heap is wiped

Strategy 1 — serialize heap state (legacy)
use ic_cdk::storage;

#[ic_cdk::pre_upgrade]
fn pre_upgrade() {
    let state = STATE.with(|s| s.borrow().clone());
    storage::stable_save((state,)).expect("Failed to save state");
}

#[ic_cdk::post_upgrade]
fn post_upgrade() {
    let (state,): (MyState,) =
        storage::stable_restore().expect("Failed to restore state");
    STATE.with(|s| *s.borrow_mut() = state);
}
Strategy 2 — stable structures (recommended)
// No pre_upgrade / post_upgrade needed for data persistence!
// StableBTreeMap, StableVec, StableCell survive automatically.

// You may still need post_upgrade for schema migrations:
#[ic_cdk::post_upgrade]
fn post_upgrade() {
    let version = CONFIG.with(|c| c.borrow().get().schema_version);
    if version < 2 {
        migrate_v1_to_v2();
        CONFIG.with(|c| {
            let mut cfg = c.borrow().get().clone();
            cfg.schema_version = 2;
            c.borrow_mut().set(cfg).unwrap();
        });
    }
    // Re-register timers (heap-side, reset on upgrade)
    start_background_timer();
}
⚠ Upgrade safety rules
  • ▸pre_upgrade can trap safely: if pre_upgrade panics, the upgrade is aborted and the old canister keeps running. Make it robust.
  • ▸post_upgrade trap = bricked canister: if post_upgrade panics, the upgrade completes but the canister can't process any messages. Always test upgrades on testnets first.
  • ▸Candid interface backwards compatibility: you can add new optional fields and methods, but removing or changing existing signatures breaks callers.
  • ▸Test with pocket-ic: simulate upgrades in tests and verify state survives before deploying to mainnet.
Principals — ICP's universal identity
A Principal is ICP's universal identifier — it identifies users, canisters, subnets, and nodes. Principals are 29-byte blobs encoded in a human-readable base32 format like rrkah-fqaaa-aaaaa-aaaaq-cai. Every canister call has an associated caller Principal. The anonymous Principal represents unauthenticated callers. Internet Identity creates derived Principals that are unique per dApp.

Internet Identity — decentralized authentication
Internet Identity (II) uses WebAuthn (device biometrics, passkeys, hardware keys) to authenticate users — no passwords, no seed phrases. II creates a different derived Principal for each dApp, preventing cross-app tracking. The frontend integrates via @dfinity/auth-client. Your backend canister receives the app-specific Principal via ic_cdk::caller() and doesn't interact with II directly.
WebAuthn / passkeys per-app derived IDs no passwords

// It never interacts with II directly

#[ic_cdk::update]
fn create_profile(name: String) -> Result<String, String> {
    let caller = ic_cdk::caller();

    // Always reject anonymous callers for write operations
    if caller == Principal::anonymous() {
        return Err("must be authenticated via Internet Identity".to_string());
    }

    // The principal IS the user's stable identity for this dApp
    // Same user + same II anchor + same dApp origin = same principal always
    PROFILES.with(|p| {
        p.borrow_mut().insert(caller, Profile {
            name,
            created_at: ic_cdk::api::time(),
        });
    });
    Ok("profile created".to_string())
}

#[ic_cdk::query]
fn get_my_profile() -> Option<Profile> {
    PROFILES.with(|p| p.borrow().get(&ic_cdk::caller()))
}

⚠ Key facts for backend developers
  • ▸Same user, different dApps = different principals: a user's II principal on dapp-a.ic0.app differs from dapp-b.ic0.app. You cannot cross-reference users across dApps.

Inter-canister Calls — async messaging between canisters
Canisters communicate by sending messages to each other. Calls are asynchronous — when canister A calls canister B, A suspends at the await point and other messages can arrive. This creates reentrancy risk: state can change between your check and your write. The solution: update state before awaiting, restore on failure (Checks-Effects-Interactions pattern).
async message passing reentrancy risk awaits enable interleaving
Making a call to another canister
use candid::Principal;
use ic_cdk::call;

#[ic_cdk::update]
async fn transfer_and_notify(to: Principal, amount: u64) -> String {
    let ledger = Principal::from_text("ryjl3-tyaaa-aaaaa-aaaba-cai").unwrap();

    // Args must be a tuple, even for a single argument
    let (block_index,): (u64,) =
        call(ledger, "icrc1_transfer", (TransferArgs { to, amount },))
            .await
            .expect("Transfer call failed");

    format!("Transferred in block {block_index}")
}
Reentrancy — the #1 inter-canister bug
// DANGEROUS — state can change between check and deduct
#[ic_cdk::update]
async fn withdraw_bad(amount: u64) {
    let bal = BALANCE.with(|b| *b.borrow());
    if bal < amount { ic_cdk::trap("insufficient"); }
    // ← another message could arrive here and also pass the check!
    call(ledger, "transfer", (amount,)).await.unwrap();  // suspends
    BALANCE.with(|b| *b.borrow_mut() -= amount);  // DOUBLE SPEND!
}

// SAFE — Checks-Effects-Interactions pattern
#[ic_cdk::update]
async fn withdraw_safe(amount: u64) {
    // 1. Deduct BEFORE the await
    BALANCE.with(|b| {
        let mut b = b.borrow_mut();
        if *b < amount { ic_cdk::trap("insufficient"); }
        *b -= amount;  // state updated atomically before await
    });
    // 2. External call — other messages see updated balance
    let result = call(ledger, "transfer", (amount,)).await;
    // 3. Restore on failure
    if result.is_err() {
        BALANCE.with(|b| *b.borrow_mut() += amount);
        ic_cdk::trap("transfer failed, balance restored");
    }
}

Guards & Access Control — protecting canister methods
Canisters receive calls from anyone on the internet. Access control is your responsibility — the IC does not enforce it. Common patterns: caller guards (specific principals only), admin pattern (configurable admin set), owner pattern (set once at init), and inspect_message (reject calls before execution to save cycles).
explicit access control trap to reject inspect_message filter

}

#[ic_cdk::update]
fn add_admin(new_admin: Principal) {
    require_admin();
    mutate_state(|s| { s.admins.insert(new_admin); });
}

#[ic_cdk::update]
fn remove_admin(admin: Principal) {
    require_admin();
    if admin == ic_cdk::caller() {
        ic_cdk::trap("cannot remove yourself");
    }
    mutate_state(|s| { s.admins.remove(&admin); });
}
inspect_message — pre-filter to save cycles
// Runs BEFORE update calls — read-only, can only accept or reject
#[ic_cdk::inspect_message]
fn inspect_message() {
    let method = ic_cdk::api::call::method_name();
    let caller = ic_cdk::caller();

    // Block anonymous callers from all write methods
    if caller == Principal::anonymous() {
        return;  // implicit reject — never call accept_message()
    }

    // Only admins can call admin_ methods
    if method.starts_with("admin_") {
        let is_admin = read_state(|s| s.admins.contains(&caller));
        if !is_admin { return; }  // reject
    }

    ic_cdk::api::call::accept_message();  // MUST explicitly accept!
}
Timers — scheduled execution in canisters
The ic_cdk_timers crate provides one-shot and recurring timers. Unlike #[heartbeat] (every consensus round), timers fire at specific intervals and can be cancelled. Timers live in heap memory — they must be re-registered in #[post_upgrade] after a canister upgrade or they will be lost.
scheduled tasks one-shot + recurring re-register on upgrade!
Timer API
// Cargo.toml: ic-cdk-timers = "0.9"
use ic_cdk_timers::{TimerId, set_timer, set_timer_interval, clear_timer};
use std::time::Duration;

// One-shot — fires once after delay
let id: TimerId = set_timer(
    Duration::from_secs(60),
    || { ic_cdk::println!("fired once"); }
);

// Recurring — fires every interval
let id: TimerId = set_timer_interval(
    Duration::from_secs(3600),  // every hour
    || { cleanup_expired_entries(); }
);

// Cancel
clear_timer(id);
Re-registering timers after upgrade
#[ic_cdk::init]
fn init() {
    start_timers();  // register on first install
}

#[ic_cdk::post_upgrade]
fn post_upgrade() {
    start_timers();  // MUST re-register — heap was wiped
}

fn start_timers() {
    set_timer_interval(Duration::from_secs(3600), || {
        cleanup_expired_entries();
    });
    set_timer_interval(Duration::from_secs(86400), || {
        generate_daily_report();
    });
}

// Store timer ID if you need to cancel it later
thread_local! {
    static CLEANUP_TIMER: RefCell<Option<TimerId>> = RefCell::new(None);
}
HTTP Outcalls — calling external APIs
Canisters can make HTTPS requests to external APIs. Because multiple subnet nodes must agree, the call is made by all replicas — responses must be deterministic or transformed to be identical. A transform function strips non-deterministic parts (timestamps, request IDs, headers) before consensus. Outcalls cost cycles and are limited in response size.
HTTPS only transform for consensus costs cycles
Making an HTTP outcall
use ic_cdk::api::management_canister::http_request::{
    HttpMethod, HttpHeader, CanisterHttpRequestArgument,
    TransformContext, HttpResponse, TransformArgs, http_request,
};

#[ic_cdk::update]
async fn fetch_price(asset: String) -> String {
    let request = CanisterHttpRequestArgument {
        url:               format!("https://api.example.com/price/{asset}"),
        method:            HttpMethod::GET,
        headers:           vec![
            HttpHeader { name: "Accept".into(), value: "application/json".into() },
        ],
        body:              None,
        max_response_bytes: Some(2048),
        transform:         Some(TransformContext::from_name("transform".into(), vec![])),
    };

    let cycles: u128 = 230_949_972_000;  // cost for 13-node subnet
    let (response,) = http_request(request, cycles).await
        .expect("HTTP request failed");

    String::from_utf8(response.body).unwrap()
}

// MUST be a #[query] — strips non-deterministic fields
#[ic_cdk::query]
fn transform(raw: TransformArgs) -> HttpResponse {
    HttpResponse {
        status:  raw.response.status.clone(),
        body:    raw.response.body.clone(),
        headers: vec![],  // strip headers — they differ between replicas
    }
}
⚠ Outcall rules
  • ▸Transform is mandatory: if the API returns timestamps or request IDs in the body, the transform function must strip them. Replicas that disagree on the response will fail consensus.
  • ▸HTTPS only: plain HTTP is not supported. The endpoint must have a valid TLS certificate.
  • ▸Cycles cost varies by subnet size: 13-node subnets cost ~230B cycles per call. Always budget for outgoing requests.
  • ▸Not available in local dfx by default: use the --enable-canister-http flag or test with pocket-ic.

// [dev-dependencies]: pocket-ic = "4"
#[cfg(test)]
mod integration {
    use pocket_ic::PocketIc;
    use candid::{Principal, encode_one, decode_one};

    #[test]
    fn test_full_canister() {
        let pic = PocketIc::new();
        let can_id = pic.create_canister();
        pic.add_cycles(can_id, 2_000_000_000_000);

        let wasm = include_bytes!(
            "../target/wasm32-unknown-unknown/release/my_canister.wasm"
        );
        pic.install_canister(can_id, wasm.to_vec(), vec![], None);

        // Query call
        let res = pic.query_call(
            can_id, Principal::anonymous(), "get_count",
            encode_one(()).unwrap()
        ).unwrap();
        let count: u64 = decode_one(&res).unwrap();
        assert_eq!(count, 0);

        // Update call
        pic.update_call(
            can_id, Principal::anonymous(), "increment",
            encode_one(()).unwrap()
        ).unwrap();

        // Test upgrade preserves state
        pic.upgrade_canister(can_id, wasm.to_vec(), vec![], None).unwrap();
        // verify count still = 1 after upgrade...

        // Advance time (for timer tests)
        pic.advance_time(std::time::Duration::from_secs(3600));
        pic.tick();  // run one consensus round
    }
}
