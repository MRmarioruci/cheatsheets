Ownership — Rust's core memory model
Every value has a single owner. When the owner's scope ends, the value is freed — no GC, no manual free(). Stack values (ints, floats, bool, char, fixed arrays) are copied on assignment. Heap values (String, Vec, Box…) are moved — the original binding becomes invalid. Clone for an explicit deep copy.
zero-cost memory mgmtno GCno dangling pointers
Move (heap types)
let s1 = String::from("hi");
let s2 = s1;  // s1 MOVED — now invalid
// println!("{s1}"); ← compile error
Copy (stack types)
let x: i32 = 5;
let y = x;  // copied — x still valid
// i32, f64, bool, char, &T → Copy

⚠ Edge cases
  • ▸Partial moves: moving out of a struct field makes the whole struct unusable. Prefer #[derive(Clone)] or use Option to take fields.
  • ▸Tuple of non-Copy: (String, i32) is moved, not copied — because one field is not Copy.
  • ▸Rc/Arc for shared ownership: multiple owners are possible with reference counting — mutation needs RefCell (Rc) or Mutex (Arc).
Borrowing — use without owning
References let you access a value without owning it. The rules: many simultaneous &T (read-only), OR exactly one &mut T (exclusive). Never both. This eliminates data races at compile time via the borrow checker. NLL (Non-Lexical Lifetimes) means borrows end at their last use, not end of block.
pass-by-referencezero-copy readsdata race prevention
Immutable &T many OK
let s = String::from("hi");
let r1 = &s; let r2 = &s; // fine
println!("{r1} {r2}");
Mutable &mut T exclusive
let mut s = String::from("hi");
let r = &mut s;
r.push_str(" world");
// no other ref while r lives

⚠ Edge cases
  • ▸Iterator invalidation: holding &v[0] while calling v.push() — Rust rejects it. The push might reallocate, making the reference dangling.
  • ▸Interior mutability: Cell<T> and RefCell<T> allow mutation through &T by moving borrow checks to runtime. Panics on violation.
  • ▸Self-referential structs: very hard — moving the struct invalidates the internal reference. Use Pin<T> or arena allocators.
Lifetimes — how long are references valid?
Every reference has a lifetime — the scope during which it is valid. Usually inferred; you annotate explicitly when a function returns a reference and the compiler can't determine which input it comes from. Lifetimes don't change how long data lives — they describe relationships between scopes. Think of 'a as a label, not a duration.
returning borrowed datastruct holding refsno dangling pointers
Lifetime annotation
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
// 'a = shorter of x's and y's lifetimes
Struct with ref
struct View<'a> {
    data: &'a str,
}
// View can't outlive its data
'static
let s: &'static str = "I live forever";
// All string literals are 'static
// Don't use as an easy fix — prefer owned
⚠ Edge cases
  • ▸'a: 'b means outlives: 'a lives at least as long as 'b. Longer lifetime is "bigger".
  • ▸HRTB — for<'a>: F: for<'a> Fn(&'a T) — closure works for any lifetime, not a specific one.
  • ▸Don't fight with 'static: sprinkling 'static everywhere usually means you need Arc or owned types instead.
Types & Variables — static, inferred, immutable-first
Rust is statically typed with powerful inference — you rarely write types explicitly. Variables are immutable by default; opt into mutation with mut. Shadowing re-declares a binding (can change type). const is inlined at every use. static has a fixed address. Numeric casts with as truncate silently — prefer TryFrom for safe conversions.

⚠ Edge cases
  • ▸Integer overflow: debug panics. Release wraps silently. Use .checked_add(), .saturating_add(), .wrapping_add() for explicit control.
  • ▸usize for indices: array/Vec indices must be usize. Negative indexing doesn't exist — use .get() for safe Option access.
Strings — &str vs String
&str is a borrowed view into UTF-8 data — lightweight, no allocation, always read-only. String is owned, heap-allocated, growable. Rule of thumb: use &str for function parameters (accepts both literals and String refs). Use String when you need to own, store, or build string data. Both are always valid UTF-8 — invalid strings are not possible in safe Rust.
UTF-8 only&str for paramsString for building

  • Zero allocation, lightweight
  • Works for literals and String slices
  • Best choice for function parameters
  • Cannot be grown or modified
  • Tied to its source's lifetime

greet(&owned_string);     // String ref — works
String — owned
  • Owned, growable, heap-allocated
  • Necessary when returning or storing strings
  • Can be mutated with mut
  • Heap allocation cost
  • Overkill for read-only parameters

s.push_str("hello");
let msg = format!("Hi, {s}!");
// use format! over + chains

Functions & Closures — first-class behavior
Last expression without semicolon is the return value. Closures capture their environment and are used heavily in iterators, async, and callbacks. The compiler infers closure types. move closures take ownership of captured values — required for threads and async tasks.
zero-cost abstractionclosures in iteratorscallbacks
Syntax
fn add(x: i32, y: i32) -> i32 { x + y }   // implicit return
fn nothing() { }                             // returns () unit
fn diverges() -> ! { panic!() }            // never returns

let add = |x, y| x + y;                     // closure, inferred types
let bump = |x| x + offset;                   // captures offset
let f = move || println!("{name}");         // move closure

⚠ Edge cases
  • ▸Returning closures: use impl Fn() for static dispatch (one type) or Box<dyn Fn()> for dynamic dispatch (any closure).
  • ▸FnMut needs mut to call: let mut f = || counter += 1; — f itself must be mut to call it.
  • ▸fn pointers: plain fn(i32) -> i32 implement all Fn traits and are useful for C interop or when you need a non-capturing callable.
Structs & Enums — algebraic data types
Structs bundle named fields. Enums define a type that can be one of several variants — each holding different data. Together they form Rust's type system backbone. Enums are vastly more powerful than C enums: each variant can carry data, making them ideal for state machines, errors, and parsed structures. match is exhaustive — adding a variant is a compile error at all match sites.
model domain datastate machinesreplace class hierarchies
Struct variants
struct User { name: String, age: u32 } // named
struct Meters(f64);         // tuple/newtype
struct Marker;              // unit — no data

impl User {
    pub fn new(name: &str, age: u32) -> Self {
        User { name: name.to_owned(), age }
    }
    pub fn greet(&self) -> String {
        format!("Hi, I'm {}", self.name)
    }
}
Enum with data
enum Shape {
    Circle(f64),
    Rect { w: f64, h: f64 },
    Point,
}
match shape {
    Shape::Circle(r)    => r * r * PI,
    Shape::Rect {w, h} => w * h,
    Shape::Point       => 0.0,

Traits — shared behavior & polymorphism
Traits define a contract of methods any implementing type must provide. impl Trait = static dispatch (zero overhead, monomorphized). dyn Trait = dynamic dispatch (vtable, runtime flexibility). Prefer static unless you need a heterogeneous collection or runtime type selection. The orphan rule prevents implementing a foreign trait for a foreign type.
polymorphismoperator overloadingcode reuse
Define & implement
trait Drawable {
    fn draw(&self);
    fn area(&self) -> f64 { 0.0 } // default
}
impl Drawable for Circle {
    fn draw(&self) { println!("○"); }
}
static vs dynamic dispatch
// impl Trait — zero cost, one type per callsite
fn render(s: &impl Drawable) { s.draw(); }

// dyn Trait — vtable, mixed types
let shapes: Vec<Box<dyn Drawable>> = vec![
    Box::new(Circle { .. }),
    Box::new(Rect { .. }),
];

Error Handling — explicit, composable, no exceptions
Errors are values. Option<T> for presence/absence (nullable replacement). Result<T,E> for success/failure. The ? operator propagates errors up the call stack ergonomically. Panics are for programmer bugs, not expected failures. Use thiserror for library error types, anyhow for application code.
no silent failures? propagationcomposable
Option<T>
name.unwrap()             // panic if None
name.unwrap_or("anon")   // safe default
name.map(|n| n.len())     // Some(n) or None
name.and_then(|n| f(n))  // chain Options
name.filter(|n| n.len()>2)
name.ok_or("missing")   // convert to Result
name.as_deref()           // Option<&str> from Option<String>
Result<T, E>
r.unwrap()               // panic on Err
r.unwrap_or(0)          // default on Err
r.expect("msg")        // panic with message
r.map(|v| v * 2)       // transform Ok
r.map_err(|e| log(e))   // transform Err
r.and_then(|v| next(v)) // chain Results
r.ok()                  // convert to Option
? operator & error crates
fn run() -> Result<String, io::Error> {
    let s = fs::read_to_string("f.txt")?; // return Err on fail
    Ok(s.trim().to_string())
}

// thiserror — library errors (generates boilerplate)
#[derive(thiserror::Error, Debug)]
enum MyErr {
    #[error("io: {0}")] Io(#[from] io::Error),
}

// anyhow — application errors (easiest, any error type)
fn run() -> anyhow::Result<()> {
    let s = fs::read_to_string("f")?; // just works
    Ok(())
}

Async / Await — concurrent I/O without blocking threads
async fn returns a Future — a lazy state machine. Nothing runs until .await or a runtime polls it. .await suspends the current task (yields the thread), letting other tasks run. No thread is blocked. Most common runtime: Tokio for servers/production, async-std as alternative. Don't use std::thread::sleep in async — use tokio::time::sleep.
web serversI/O concurrencythousands of connections
Basics
// Cargo.toml: tokio = { version = "1", features = ["full"] }
#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let body = fetch("https://example.com").await?;
    Ok(())
}
async fn fetch(url: &str) -> anyhow::Result<String> {
    let text = reqwest::get(url).await?.text().await?;
    Ok(text)
}
Concurrent tasks
// spawn — independent tasks
let h = tokio::spawn(async { work().await });
let result = h.await??;

// join! — all complete
let (a, b) = tokio::join!(fut_a, fut_b);

// select! — first wins
tokio::select! {
    v = fut_a => { },
    _ = time::sleep(dur) => { },
}
Shared state
use std::sync::Arc;
use tokio::sync::Mutex; // async mutex

let state = Arc::new(Mutex::new(0));
let s = state.clone();
tokio::spawn(async move {
    let mut lock = s.lock().await;
    *lock += 1;
});

Generics — write once, specialize at compile time
Generics are monomorphized — the compiler generates a concrete version for each type used. Zero runtime cost. Trait bounds constrain accepted types. Three styles for trait bounds are equivalent; use where clauses for readability with multiple/complex bounds.
zero-cost abstractionDRY across typescompile-time specialization
Generic syntax styles
// All equivalent — pick based on readability
fn f<T: Display + Clone>(x: T) { }
fn f(x: impl Display + Clone) { }
fn f<T>(x: T) where T: Display + Clone { }

// Generic struct
struct Stack<T> { items: Vec<T> }
impl<T> Stack<T> { /* any T */ }
impl<T: Display> Stack<T> { /* only when T: Display */ }
⚠ Edge cases
  • ▸Monomorphization bloat: many different type uses means larger binary. Use dyn Trait to trade runtime cost for size.
  • ▸Turbofish: "42".parse::<i32>() — when the compiler can't infer the type, annotate with ::<Type>.
  • ▸PhantomData: for zero-size type tracking without holding a T value — common in unsafe code and type-state patterns.

Smart pointers are structs that behave like pointers but add capabilities: heap allocation, reference counting, interior mutability, or thread safety. RefCell<T> — interior mutability (single-thread)
use std::cell::RefCell;
let data = RefCell::new(vec![1,2,3]);

data.borrow();       // &Vec — panics if mut borrow exists
data.borrow_mut();   // &mut Vec — panics if any borrow exists
data.try_borrow();   // Result — safe, no panic

// Common pattern: Rc<RefCell<T>> = shared mutable (single-thread)
let shared = Rc::new(RefCell::new(vec![]));
shared.borrow_mut().push(1);
Mutex<T> — shared mutable across threads
use std::sync::{Arc, Mutex};
let counter = Arc::new(Mutex::new(0));

let c = counter.clone();
std::thread::spawn(move || {
    let mut n = c.lock().unwrap(); // blocks until available
    *n += 1;
}); // lock released when n drops
// Use tokio::sync::Mutex in async code!
⚠ Edge cases
  • ▸Rc cycles leak memory: Rc<T> doesn't handle cycles — use Weak<T> (a non-owning reference) to break cycles. Rc::downgrade() creates a Weak.
  • ▸Mutex poisoning: if a thread panics while holding a lock, the Mutex is "poisoned". lock().unwrap() panics — use lock().unwrap_or_else(|e| e.into_inner()) to recover.
  • ▸std Mutex in async: never hold a std::sync::Mutex across an .await — use tokio::sync::Mutex instead to avoid deadlocks.
  • ▸Cow for efficient strings: Cow<'_, str> is either &str (borrowed) or String (owned). Great for functions that sometimes need to modify and sometimes don't.

Project Structure — main.rs vs lib.rs, workspaces, layout
A crate is Rust's compilation unit. main.rs is the root of a binary crate — it has a main() and produces an executable. lib.rs is the root of a library crate — no main(), produces a .rlib for others to use. A package can have both: one lib.rs and multiple binary crates in src/bin/. Workspaces group multiple crates that share a Cargo.lock and dependency cache.
binary vs librarymodule organizationworkspaces
main.rs — binary crate
  • Entry point: fn main()
  • Produces an executable
  • cargo run works directly
  • Can depend on a lib.rs in same package
  • Cannot be imported by other crates
  • Tests are more limited
// src/main.rs
fn main() {
    println!("Hello!");
}
// build: cargo build
// run:   cargo run
// test:  cargo test
lib.rs — library crate
  • No main() function
  • Produces a .rlib importable by others
  • Published to crates.io
  • Full integration test support in tests/
  • Public API documented with //!
  • Cannot be run directly
// src/lib.rs
//! Crate-level documentation here
pub mod math;
pub use math::add; // re-export

// importers: use mycrate::add;
Standard project layout
my_project/ ├── Cargo.toml ← package manifest ├── Cargo.lock ← locked dependency versions (commit this!) ├── .gitignore ├── src/ │ ├── main.rs ← binary entry point │ ├── lib.rs ← library root (optional) │ ├── config.rs ← module (mod config; in lib.rs) │ ├── models/ │ │ ├── mod.rs ← old style, OR models.rs at parent level │ │ └── user.rs │ └── bin/ │ └── cli.rs ← extra binary: cargo run --bin cli ├── tests/ │ └── integration.rs ← integration tests (use lib as extern) ├── benches/ │ └── bench.rs ← benchmarks (use criterion crate) ├── examples/ │ └── demo.rs ← cargo run --example demo └── build.rs ← build script (code gen, linking)

Idioms & Patterns — writing idiomatic Rust
Idiomatic Rust avoids common anti-patterns from other languages. These patterns appear constantly in production code. Learning them makes code cleaner, more performant, and more readable to other Rust developers.
idiomatic codecommon patternsavoid anti-patterns
Builder pattern
Construct complex objects step-by-step without many constructor arguments. Each method returns self for chaining.
struct Request { url: String, timeout: u64, retry: bool }
struct RequestBuilder { url: String, timeout: u64, retry: bool }

impl RequestBuilder {
    fn new(url: &str) -> Self { Self { url: url.into(), timeout: 30, retry: false } }
    fn timeout(mut self, t: u64) -> Self { self.timeout = t; self }
    fn retry(mut self) -> Self { self.retry = true; self }
    fn build(self) -> Request { Request { url: self.url, timeout: self.timeout, retry: self.retry } }
}
let req = RequestBuilder::new("https://api.com").timeout(60).retry().build();
Newtype pattern
Wrap a primitive in a struct to create a distinct type. Prevents mixing up values of the same underlying type (e.g., Meters vs Kilograms).
struct Meters(f64);
struct Kilograms(f64);

fn calc(dist: Meters, mass: Kilograms) { }
// calc(Kilograms(5.0), Meters(10.0)); ← compile error! types swapped

// Also used to implement foreign traits on foreign types:
struct Wrapper(Vec<String>);
impl std::fmt::Display for Wrapper { /* implement Display for Vec */ }
Type-state pattern
Encode state in the type system so invalid transitions are compile errors. A connection can't call query() unless it's already Connected.
struct Connected;
struct Disconnected;

struct Conn<State> { _state: std::marker::PhantomData<State> }

impl Conn<Disconnected> {
    fn connect(self) -> Conn<Connected> { /* ... */ todo!() }
}
impl Conn<Connected> {
    fn query(&self) -> String { todo!() }
}
// Conn<Disconnected>::new().query(); ← compile error!
